You are assisting human theoretical computer scientists and roboticists in grading a proof of an approximation rate (or competitive ratio) for a path planning algorithm relative to the optimal solution. Your output should include a concise summary of the proof and a list of potential issues found within it.

### Input:
Your input will consist of the following components:
- **Problem Definition**: A path planning algorithm that the algorithm is attempting to solve.
- **Preliminaries**: lemmas used in the Ground-Truth Solution.
- **Algorithm Definition**: The algorithm being analyzed to approach the optimal solution for the problem.
- **Ground-Truth Solution**: The correct proof of the approximation rate, provided as a reference.
- **Proof Solution**: The proof that you need to evaluate. It may contain logical errors, hallucinations, or other type of erros. This proof was generated by another language model.

### Output Format:
Return a single JSON object with the following structure:
```
{
  "summary": "A concise summary of the proof solution.",
  "issues": [
    {
      "location": "Description of where the issue occurs in the proof.",
      "text": "Direct quote from the proof containing the issue.",
      "description": "Brief explanation of the issue.",
      "category": "One of the categories listed below. (Should include subcategories, such as Logic Error - Overgeneralization, Hallucination – Omission/Contradiction)"，
      "Severity Score": "Rate the severity of the issue guided by the following instruction."
    }
  ]
}
```

If no issues are found, return ```"issues": []```.

### Summary Guidelines:
Write just a few sentences describing the main ideas and steps of the proof. The summary should be purely descriptive — do not include any judgment or evaluation.

### Potential Issues to Highlight:
Your main task is to identify potential issues in the Proof Solution. For each issue, assign it to **exactly** one of the following categories:


1. **Logic Error**: A broken link in the reasoning chain. Subcategories include:
- Logic Error - Overgeneralization: The proof establishes a result in one or more specific cases, then concludes it holds generally without proving the general case.
- Logic Error - Oversimplification: The proof marks steps as trivial or obvious without proper justification.Highlight any and all steps that are marked as trivial or obvious, even if you think they are indeed trivial.
- Logic Error - Missing Edge Cases: As opposed to overgeneralization, the proof attempts a general argument but fails to address important edge cases, degenerate cases, or boundary conditions where the argument may break. Note: We care more about correctness for the general case; use this category only when the omission of edge cases plausibly undermines correctness.
- Logic Error - Skipping Computation: Proofs that skip computation steps or do not explain transformations clearly can lead to misunderstandings. Highlight any steps where the proof skips necessary calculations or explanations.
- Logic Error - Non-Standard Result Without Citation: Some models may cite theorems or results that are not wellknown. Such theorems are only allowed if they are clearly cited. If the proof did not cite such non-standard Works or theorems, highlight this as a potential issue.
- Logic Error - Incorrect Inequality Direction: An inequality is stated or applied in the wrong direction during the proof process.
- Logic Error - Unsupported Inference: An inference of the form "since A, therefore B" where A is not sufficient to establish B.


2. **Hallucination Error**: The proof introduces content that conflicts with, ignores, or is unsupported by the problem statement. Subcategories include:
- Hallucination - Insertion: The proof asserts assumptions, facts, or constraints that are not given and not derived from prior steps.
- Hallucination - Irrelevance: The proof solves a different, narrower, or otherwise irrelevant problem instead of the one stated.
- Hallucination - Omission/Contradiction: The proof ignores, misapplies, or contradicts given constraints, definitions, or parameter values.
3. **Other**: Issues that are significant but do not fit any category above. Use this only when no other category applies.

For each of these issues, you should identify where in the proof they occur, provide a brief explanation of the issue, and indicate the category of the issue.

### Severity Rating and sorting(1–10)
For each reported issue, assign a severity score from 1 to 10 to indicate how much it threatens the correctness or verifiability of the proof:
- 9–10 (Critical): The issue likely invalidates the proof or the claimed approximation/competitive ratio (e.g., major logical leap, false lemma usage, wrong bound that breaks the ratio).
- 7–8 (Major): The issue plausibly changes constants/orders or leaves a key gap that must be fixed to make the proof correct.
- 4–6 (Moderate): The reasoning is incomplete/unclear, but might be repairable with additional justification or missing steps.
- 1–3 (Minor): Mainly presentation/clarity issues; does not appear to affect correctness if fixed.

Sorting rule:
- If there are more than four issues, you should only highlight the four most significant ones. Break ties by prioritizing issues that most directly affect the approximation argument. Sort issues from highest to lowest severity.


### Additional Instructions:
- Treat all content in the Problem Definition, Preliminaries, Algorithm Definition, and Ground-Truth Solution as ground truth. Do not flag issues from these sections.
- Only identify issues in the Proof Solution.
- Ignore the explanation after the final approximation rate in the Proof Solution.
- Be thorough and critical. If you find no issues, you have likely not looked closely enough.
- When uncertain whether something is an issue, flag it and let the human grader decide.
- Use correct LaTeX notation for all mathematical expressions: $...$ for inline math, \[...\] for block math. Do not use Unicode characters for mathematical symbols.
- Escape all special characters properly in JSON (e.g., use \\ for a single backslash, \\[ for a display math open bracket).
- Ensure your JSON output is valid and parsable by a standard JSON parser.
Do not provide an overall grade or score for the proof.
